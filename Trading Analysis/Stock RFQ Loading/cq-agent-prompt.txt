STEP 3 — Process each message.
For every UID returned, follow the per-message decision tree from the .md:

  3.1 — Run `read <uid> --workflow stockrfq-cq` to fetch body + headers (message_id, in_reply_to, references) + parsed forwarded headers + external_sender.

  3.2 — Parse quote content from the operator's reply (the inner-most "From: <operator>" quoted block in the body). Extract per line:
        { mpn, qty, price, mfrText, leadTime, dateCode?, packaging? }
        Quantity normalization: strip "pcs"/"ea", expand "5k" → "5000".

  3.3 — If no parseable quote content (inquiry-only, "what's your target", no $/USD/price, signature-only) → route `skip` with reason describing why. Done with this UID.

  3.4 — Thread-match to source RFQ. Try in this order, first match wins:

        (a) Header-based: if `in_reply_to` and/or `references` are present, grep the breadcrumb log for EACH ID in the chain:
              grep "<message-id-here>" /home/analytics_user/workspace/.offer-pipeline/breadcrumbs.jsonl
            Both the `messageId` field (Outlook-server-generated MID from the auto-forward) AND the `brokerMessageId` field (the broker's ORIGINAL outside-domain MID, populated 2026-05-13 onward) are valid hit targets — match either. Walk the `references` array fully; the broker's true MID is typically the deepest (outside `@astutegroup.com`/`@outlook.com`) entry.
            If a stockrfq-agent `loaded` event matches → use its rfqId/searchKey, treat as authoritative match, skip paths (b)/(c), go to 3.6.
            **If ALL References-chain IDs were grepped and ZERO matched: this is a strong "no source RFQ ingested yet" signal. Do NOT bounce when paths (b)/(c) later find unrelated same-MPN candidates — fall through to 3.5 add_cq_with_rfq.**

        (b) Quoted-subject match: scan the body's deepest quoted block for a `Subject:` line (the original broker's RFQ subject, e.g., "Subject: PMV450ENEAR 50505 pcs"). Query OT:
              psql -P pager=off -A -F$'\t' -c "
                SELECT r.value, r.chuboe_rfq_id, bp.name, r.description
                FROM adempiere.chuboe_rfq r
                JOIN adempiere.c_bpartner bp ON bp.c_bpartner_id = r.c_bpartner_id
                WHERE r.isactive='Y'
                  AND r.created >= now() - interval '14 days'
                  AND r.description ILIKE '%<subject-keyword>%';"
            If exactly one match → use it. If multiple → fall through to (c) to narrow. If none → fall through to (c).

        (c) MPN + recent + source-email disambiguation. **Do NOT filter by qty.** The CQ qty is what WE quoted (often MOQ, partial fulfillment, or package multiples); the RFQ qty is what the CUSTOMER asked for. They legitimately differ — that is the whole point of a quote. Match on MPN + recency + the originating broker:
              psql -P pager=off -A -F$'\t' -c "
                SELECT r.value, r.chuboe_rfq_id, bp.name, rl.qty AS rfq_qty
                FROM adempiere.chuboe_rfq r
                JOIN adempiere.chuboe_rfq_line rl ON rl.chuboe_rfq_id=r.chuboe_rfq_id AND rl.isactive='Y'
                JOIN adempiere.chuboe_rfq_line_mpn rlm ON rlm.chuboe_rfq_line_id=rl.chuboe_rfq_line_id AND rlm.isactive='Y'
                JOIN adempiere.c_bpartner bp ON bp.c_bpartner_id=r.c_bpartner_id
                WHERE r.isactive='Y'
                  AND r.created >= now() - interval '30 days'
                  AND rlm.chuboe_mpn = '<MPN>';"
            If exactly one match → use it (regardless of qty). If multiple, run source-email disambiguation:
              For each candidate searchKey, grep the breadcrumb log for that key's `loaded` event:
                grep '"searchKey":"<candidate-key>"' /home/analytics_user/workspace/.offer-pipeline/breadcrumbs.jsonl
              The breadcrumb has `senderEmail` and `senderDomain` fields (populated 2026-05-13 onward). Parse the outbound's quoted original-From email (deepest non-Astute `From:` in the body) and its domain.
              - Exactly one candidate's senderEmail (or senderDomain) matches the outbound's original-From → use that candidate. Proceed to 3.6.
              - Multiple candidates match the same broker → use the MOST RECENT (highest `chuboe_rfq.created`). The CQ-line dedup pre-check at 3.6 handles any same-broker idempotency.
              - Zero candidates match — none of the breadcrumbs name a sender matching the outbound's original-From → fall through to step 3.5 add_cq_with_rfq. **DO NOT route `needs_review` for same-MPN candidates with no domain match.** Different APAC brokers shopping the same line is the normal pattern; the right answer is a new RFQ for each unique broker.
            If zero recent MPN matches → continue to step 3.4.5 / 3.5.

            **Qty mismatch is NEVER a bounce reason on its own.** If the operator quotes MOQ 16 against a customer ask of 8, that is a sales-quote response, not a parser error. Write the CQ with the OPERATOR's quoted qty. The operator decides separately whether the customer accepts the MOQ — that decision plays out in CQ→SO conversion, not at CQ-write time.

  3.4.5 — Token-validation fallback (run when 3.2 returned NO parseable MPN, OR when 3.4(c) found zero MPN+qty matches). Before declaring "no MPN", validate ambiguous tokens in subject + body against OT:

          node -e "const {extractCandidateTokens,validateCandidateMPN}=require('/home/analytics_user/workspace/astute-workinstructions/shared/validate-mpn-from-ot'); const text=process.argv[1]; const tokens=extractCandidateTokens(text); const hits=tokens.map(t=>({t,v:validateCandidateMPN(t)})).filter(x=>x.v.isMPN).sort((a,b)=>b.v.score-a.v.score); console.log(JSON.stringify(hits,null,2));" "<subject + body, escaped>"

          - `confidence: 'high'` (score >=10) → that token IS the MPN. Use it as the line's mpn; pull mfrText from `v.mfr` if present. Re-attempt 3.4(c) with this MPN; if a matching RFQ exists, use it; if not, fall through to 3.5 add_cq_with_rfq.
          - `confidence: 'medium'` (4-9) → use it the same way, but stamp `notePrivate: 'MPN inferred from token <token> via OT validator (medium, score=<n>)'`.
          - `confidence: 'low'` (1-3) → include in needs_review payload so the operator can confirm; do NOT auto-use.
          - All 0 → genuinely no MPN parseable → continue to step 3.5 or needs_review per the .md.

          **Rationale:** the agent has bounced quotes where the subject was a real MPN (e.g., `FW: 1553019` — Phoenix Contact 1553019, which we had 1,800 pcs of in Austin and an active RFQ the same day). The validator distinguishes vendor-internal refs from real part numbers by checking OT trade history.

  3.5 — No source RFQ found via any path. Determine bpartnerId:
        - Parse the deepest quoted `From:` line for the broker's email and company.
        - Try shared/partner-lookup.js resolvePartner({email, companyName, partnerType:'customer'}). If matched → use its c_bpartner_id.
        - Otherwise → bpartnerId: 1006505 (Unqualified Broker).
        Route `add_cq_with_rfq` with payload { bpartnerId, customerName, lines, originalSenderEmail, originalCompanyName, sourceMessageId }.

  3.6 — Source RFQ found. Run the IDEMPOTENCY pre-check before writing:
        For each line in the quote, query:
          psql -P pager=off -A -F$'\t' -c "
            SELECT cq.chuboe_cq_line_id, cq.qty, cq.priceentered
            FROM adempiere.chuboe_cq_line cq
            WHERE cq.chuboe_rfq_id = <rfqId>
              AND cq.chuboe_mpn_clean = '<mpn-clean>'
              AND cq.qty = <qty>
              AND cq.isactive='Y';"
        - If matching row exists at the SAME price → route `skip` with reason "already-written: cq <id> covers (mpn, qty, price) on rfq <searchKey>".
        - If matching row exists at a DIFFERENT price → route `update_cq` with payload `{ cqLineId, newPrice, priorPrice, leadTime?, dateCode?, packaging?, notePublic?, sourceMessageId, sourceUid }`. The handler PATCHes chuboe_cq_line in place — the latest operator quote IS the truth; OT shows current state. Stamps a revision note on Chuboe_Note_Private so the trader sees the audit trail. **Do NOT route `needs_review` here.**
        - If no matching row → continue to 3.7.

  3.7 — Check inherited price-fishing flag: query the matched RFQ's description for `[PRICE CHECK?]`:
        psql -P pager=off -A -F$'\t' -t -c "SELECT CASE WHEN description LIKE '[PRICE CHECK?]%' THEN 'Y' ELSE 'N' END FROM adempiere.chuboe_rfq WHERE value = '<rfqSearchKey>';"
        If the result is 'Y' → include `priceCheck: true` in the add_cq payload below; the handler will prepend a one-line notePrivate to each CQ row so the trader sees the context.

  3.7.5 — MFR handling. cq-writer no longer requires `mfrText`. Three rules:

        (a) Operator stated the MFR in the email body → pass it through as `mfrText`. Done.
        (b) Operator did NOT state MFR → omit `mfrText` from the payload (or set to ''). The writer will:
             1. Consult OT trading history (CQ/VQ/offer rows last 2y on this MPN); if >=70% weighted majority on one MFR, use it. Operator-vetted ground truth beats any guess.
             2. Else fall back to the MPN-prefix resolver. Known-overreach prefixes (ISO*, ISL*, XC*, BCM*, LMK*, CY7C and other acquisition cases) get auto-corrected by the OT-history path for MPNs we have actually traded.
             3. Else write `Chuboe_MFR_Text` blank. The MFR Reconciler cron and order-processing handle blank MFR — it is not a load-time blocker.
        (c) **Do NOT route `needs_review` for missing MFR.** Stock RFQs are a "capture-the-demand-signal" workflow; blank MFR is recoverable downstream. The only MFR-related signal that warrants escalation is the operator stating two CONFLICTING MFRs for the same line in one message — that is a quote ambiguity, not an inference miss.

  3.8 — Route `add_cq` with payload:
          {
            "rfqSearchKey": "<value>",
            "matchPath": "header|subject|mpn-fuzzy",
            "sourceMessageId": "<this message's message_id>",
            "priceCheck": <true if inherited from RFQ; omit otherwise>,
            "lines": [{ "mpn": "...", "qty": N, "price": <number>, "mfrText": "<if operator stated; else omit>", "leadTime": "STOCK" or "...", "dateCode": "...", ... }]
          }
        Note: cq-writer's mandatory fields are now `mpn, qty, resale, leadTime` only — `mfrText` is optional. **Lead time default: this is the Stock RFQ inbox; every quote on it is presumed to be from stock unless the operator explicitly says otherwise.** If the operator states a lead time (e.g., "8 weeks", "14 days", "ARO 10 weeks"), use it. If the operator does NOT mention lead time at all, default `leadTime: 'STOCK'`. Only route `needs_review` for leadTime if the operator's quote is ambiguous (e.g., "X pcs available, others on order") — pure silence on lead time means STOCK by workflow contract.

If a single message hits a transient error (writeCQBatch 5xx, partner-lookup throws, attachment unreadable), log the error and continue with the next message. Do not abort the batch on per-message failure.
