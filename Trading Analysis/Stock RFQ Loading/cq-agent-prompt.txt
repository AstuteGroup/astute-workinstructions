  3.2 — Parse quote content from the operator's reply (the inner-most "From: <operator>" quoted block in the body). Extract per line:

  3.3 — If no parseable quote content (inquiry-only, "what's your target", no $/USD/price, signature-only) → route `skip` with reason describing why. Done with this UID.

  3.4 — Thread-match to source RFQ. Try in this order, first match wins:

              grep "<message-id-here>" /home/analytics_user/workspace/.offer-pipeline/breadcrumbs.jsonl
            If a stockrfq-agent `loaded` event matches → use its rfqId/searchKey. Walk `references` array as fallback (the chain may include the original broker's Message-ID even if In-Reply-To points at an intermediate Astute reply).

        (b) Quoted-subject match: scan the body's deepest quoted block for a `Subject:` line (the original broker's RFQ subject, e.g., "Subject: PMV450ENEAR 50505 pcs"). Query OT:
              psql -P pager=off -A -F$'\t' -c "
                SELECT r.value, r.chuboe_rfq_id, bp.name, r.description
                FROM adempiere.chuboe_rfq r
                JOIN adempiere.c_bpartner bp ON bp.c_bpartner_id = r.c_bpartner_id
                WHERE r.isactive='Y'
                  AND r.created >= now() - interval '14 days'
                  AND r.description ILIKE '%<subject-keyword>%';"
            If exactly one match → use it. If multiple → fall through to (c) to narrow. If none → fall through to (c).

        (c) MPN + qty + recent + customer-domain fuzzy:
              psql -P pager=off -A -F$'\t' -c "
                SELECT r.value, r.chuboe_rfq_id, bp.name, rl.qty
                FROM adempiere.chuboe_rfq r
                JOIN adempiere.chuboe_rfq_line rl ON rl.chuboe_rfq_id=r.chuboe_rfq_id AND rl.isactive='Y'
                JOIN adempiere.chuboe_rfq_line_mpn rlm ON rlm.chuboe_rfq_line_id=rl.chuboe_rfq_line_id AND rlm.isactive='Y'
                JOIN adempiere.c_bpartner bp ON bp.c_bpartner_id=r.c_bpartner_id

        - If matching row exists at a DIFFERENT price → route `needs_review` with reason "existing cq <id> has different price (<old> vs <new>); operator may have revised the quote — do not auto-supersede".
        - If no matching row → continue to 3.7.

  3.7 — Check inherited price-fishing flag: query the matched RFQ's description for `[PRICE CHECK?]`:

        If the result is 'Y' → include `priceCheck: true` in the add_cq payload below; the handler will prepend a one-line notePrivate to each CQ row so the trader sees the context.

  3.8 — Route `add_cq` with payload:
