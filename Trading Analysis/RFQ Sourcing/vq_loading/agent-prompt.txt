3.6 — RFQ search key resolution.
────────────────────────────────────────────

The agent must find the right RFQ for these quotes. In priority order:

  1. **Subject line carries the RFQ #** — common for Type 2 (`FW: 1132932`, `RE: RFQ 1133012`). Extract the 7-digit number directly.
  2. **Body mentions the RFQ #** — look for `RFQ#` / `RFQ:` / `rfq number is` patterns.
  3. **MPN match against ALL active RFQs** — when no explicit RFQ ID. No age cutoff on the query itself; the 30-day window is only used to decide multi-write fan-out (see step 3.6.b). Run TWO passes; second only if first returns nothing:

     **Pass 1 — exact match on chuboe_mpn_clean** (normalization already applied — hyphens/spaces stripped, uppercased):
       SELECT r.value, r.created, rl.chuboe_rfq_line_id, rl.line, lm.chuboe_mpn, lm.chuboe_mpn_clean, rl.qty
       FROM adempiere.chuboe_rfq r
       JOIN adempiere.chuboe_rfq_line rl ON r.chuboe_rfq_id=rl.chuboe_rfq_id AND rl.isactive='Y'
       JOIN adempiere.chuboe_rfq_line_mpn lm ON rl.chuboe_rfq_line_id=lm.chuboe_rfq_line_id AND lm.isactive='Y'
       WHERE r.isactive='Y'
         AND UPPER(lm.chuboe_mpn_clean) IN (<normalized vendor MPNs — strip non-alphanumeric, uppercase>)
       ORDER BY r.created DESC;

     **Pass 2 — prefix/suffix fuzzy match** (only if Pass 1 returned 0 rows):
       Mirrors loadBulkSummary.matchMpnToLine: RFQ's normalized MPN is a prefix of the vendor's normalized MPN, or vice versa, with min 8 chars overlap. Use one query per extracted MPN:
       SELECT r.value, r.created, rl.chuboe_rfq_line_id, rl.line, lm.chuboe_mpn, lm.chuboe_mpn_clean, rl.qty
       FROM adempiere.chuboe_rfq r
       JOIN adempiere.chuboe_rfq_line rl ON r.chuboe_rfq_id=rl.chuboe_rfq_id AND rl.isactive='Y'
       JOIN adempiere.chuboe_rfq_line_mpn lm ON rl.chuboe_rfq_line_id=lm.chuboe_rfq_line_id AND lm.isactive='Y'
       WHERE r.isactive='Y'
         AND LENGTH(lm.chuboe_mpn_clean) >= 8
         AND (UPPER(lm.chuboe_mpn_clean) LIKE UPPER('<vendor MPN normalized>') || '%'
              OR UPPER('<vendor MPN normalized>') LIKE UPPER(lm.chuboe_mpn_clean) || '%')
       ORDER BY r.created DESC;

     Concrete example (UID 8366 calibration): vendor quoted `BTS50162EKAXUMA1` (the cut-tape variant with `XUMA1` suffix). RFQ 1132427 has `BTS5016-2EKA` / clean `BTS50162EKA`. Pass 1 misses; Pass 2 matches because `BTS50162EKAXUMA1` starts with `BTS50162EKA`. The match is the customer's RFQ MPN — the vendor's full MPN goes into `vendorQuotedMpn` per the .md Alternate MPN rule.

  4. **Multi-write fan-out rule** — decide which RFQ(s) get the VQ:

     Let `matches` = all active RFQs returned by Pass 1 or Pass 2 above (deduplicated by `r.value`), ordered by `r.created DESC`.

     - **`len(matches) == 0`** → no active RFQ for this MPN. Route `need_info_vendor` with `{missing: ['rfq_number'], extracted: {<partial quotes>}, subject, outerFrom}`. Sidecar waits for Jake to clarify which RFQ.
     - **`len(matches) == 1`** → write to that single RFQ. Age is irrelevant (could be 3 days or 6 months — single match means no ambiguity to resolve).
     - **`len(matches) > 1` AND at least one match has `created >= NOW() - INTERVAL '30 days'`** → write to ALL matches inside the 30-day window. Set `rfqSearchKey` to the most recent of those, `secondaryRfqSearchKeys` to the rest within the window. RFQs older than 30 days are ignored.
     - **`len(matches) > 1` AND no match is within 30 days** → write only to the single most-recent match (`rfqSearchKey` = `matches[0].value`, no secondaries). "For reference" — the older RFQs may still be open and the buyer can use the price/availability signal.

     Qty alignment is NOT a gate. If a fan-out RFQ asks 10 and the broker quotes 1815, write the VQ anyway. Buyers and sellers use the price/availability data regardless of qty match.
  4. **Ambiguous** (multiple candidate RFQs, no single covers all) → route `needs_review` with `{reason: "RFQ resolution ambiguous", details: "<candidate RFQs>"}`.
  5. **None found** → route `need_info_vendor` with `{missing: ["rfq_number"], extracted: {<partial quotes>}, subject, outerFrom}`.

Validate the RFQ exists and is active:
  SELECT chuboe_rfq_id, value, created FROM adempiere.chuboe_rfq
  WHERE value='<rfqSearchKey>' AND isactive='Y';

**If the subject-cited or body-cited RFQ is INACTIVE (returns 0 rows above), do NOT bounce.** Fall through to step 3.6.3 (MPN matching against ALL active RFQs) — the cited RFQ may have been deactivated and replaced. The MPN match will find the live RFQ. Treat the cited number as a hint that lost authority, not a load target.

**If the subject-cited RFQ is active AND MPN matching also finds different active RFQs:** the subject is the authoritative target (operator's explicit reference). Use it as the primary. Apply multi-write fan-out (step 3.6.4) only if MPN matching surfaces ADDITIONAL active RFQs within 30 days — those become secondaries.

────────────────────────────────────────────
3.7.5 — EXHAUST RECOVERABLE SIGNALS before bouncing.
────────────────────────────────────────────

**Discipline (mirrors stockrfq-cq's exhaust-recoverable-signals pattern, per memory feedback_cq_exhaust_recoverable_signals.md).** Before routing `needs_review` or `need_info_vendor`, try harder. Premature surrender is the failure mode that turned 4 of 4 recovery messages into needs_review on the 13:00 calibration tick.

Apply these recovery steps in order. Each one is a "before you bounce, try this" gate:

  (a) **Missing MFR → infer from MPN prefix WITH SANITY CHECK.** If the extracted quote has `mfr` blank or ambiguous, call `shared/mfr-resolver.resolveMfrForRow({mpn: <vendor MPN>})` synchronously. Returns `{matched, canonical, id, ...}`.

       **⚠️ KNOWN RESOLVER OVERREACH (2026-05-14, per project_mfr_resolver_prefix_overreach.md):** the resolver has a ~7% misclassification rate on certain prefixes. Confirmed bad mappings:
       - `ISO*` → resolver says Issi; many are actually TI (ISO7821, ISO5851 isolation/digital)
       - `ISL*` → resolver says Issi; most are actually Renesas (Intersil acquired)
       - `XC*` → resolver says AMD/Xilinx; many are Torex (XC6219, XC9258 LDOs)
       - `BCM*` → resolver may say Broadcom when the part is Nexperia (BCM857 transistors) or others
       - `LMK*` → can confuse TI vs SiTime
       Treat resolver output as a HINT, not source-of-truth. Apply this sanity check:

       - **Step 1:** Call the resolver. If `matched: false` → flag the missing prefix as a `shared/data/mpn-prefixes.json` data-file gap (file as a deferred-work item, NOT recurring manual triage) and bounce to needs_review only if the operator absolutely cannot proceed.
       - **Step 2 (sanity check on `matched: true`):** does the resolved MFR make sense given the email context? Check:
            (i) Does the broker's signature / surrounding email mention a different MFR for this part?
            (ii) Is the resolved MFR plausible for the part category? (e.g., XC6219 is a small LDO — Torex makes those, AMD/Xilinx doesn't.)
            (iii) Has the operator previously corrected this prefix? Check the deferred-work tracker.
       - **Step 3 (decision):**
            - Sanity check passes → use `canonical` as the quote's `mfr`. Stamp `vendorNotes: 'MFR inferred from MPN prefix via mfr-resolver (HINT — verify before purchase); vendor did not specify'`.
            - Sanity check is uncertain → write the VQ with `mfr` BLANK (the MFR Reconciler cron can fill it overnight; better than a silently wrong FK). Stamp `vendorNotes: 'MFR inferred candidate from resolver: <canonical> — flagged for review (resolver prefix overreach risk)'`.
            - Sanity check clearly fails → write the VQ with `mfr` BLANK and stamp `vendorNotes: 'MFR resolver returned <canonical> but inconsistent with part class; left blank for manual resolution'`. Optionally route a separate `needs_review` if the failure is severe.

       **Principle:** silent wrong MFRs propagate downstream (wrong supplier picks, wrong cross-refs, wrong reconciliation). A blank MFR is recoverable by the reconciler; a wrong one is a latent data corruption.

  (b) **Multi-RFQ ambiguity is not ambiguity — it's fan-out.** Same MPN appearing on multiple active RFQs is the normal pattern (different customers asking for the same part). Per step 3.6.4's multi-write rule, write to ALL matches inside the 30-day window. Never route this to `needs_review` — that was the 13:00 failure mode for UIDs 8413/8414/8415.

  (c) **netCOMPONENTS URL → source RFQ hint.** When the body contains a `netcomponents.com/<long-id>` URL (visible in the original outbound RFQ block at the bottom of the forward chain), the broker is responding to a specific Astute-sent RFQ. The mapping from that URL to an OT chuboe_rfq isn't deterministic today (data gap — flagged on the roadmap), but treat the URL's presence as a strong signal that ONE specific RFQ is the right target. If MPN-match returns multiple RFQs and a netCOMPONENTS URL is present, log the URL in the route output's notes so the operator can map it back later.

  (d) **"In stock" without explicit DC → leadTime: "stock", dateCode: null.** Many brokers say "in stock" or "stock" without specifying a date code. That's NOT a need_info_vendor case — write the VQ with `leadTime: "stock"` and leave `dateCode` blank. The buyer reviews DC at purchase time.

  (e) **Currency detection — extract and set `currency` field.** ALWAYS check the quote for non-USD currency markers before treating cost as USD. Markers include:
       - Currency symbol in the cost line: `€`, `£`, `¥`, `CHF`, `¥` Chinese RMB
       - ISO code in the cost line: `EUR 12.50`, `15.00 EUR`, `GBP 9.99`
       - Sender or signature region hints: German distributors (`@schukat.com`, `GmbH`), UK distributors (`@solsta.co.uk`, `Ltd`), EU domains (`.de`, `.fr`, `.nl`, `.eu`), Asian distributors with local-currency invoicing
       - Price-looks-wrong heuristic: cost >>$100 for what's clearly a passive component, or <<$0.01 for what's a chip — strong currency-mismatch signal

     **Action:** set `currency` on the quote object to the appropriate ISO code (`"EUR"`, `"GBP"`, `"JPY"`, etc.). load-bulk-summary's `CURRENCY_MAP` covers 17 currencies (USD, EUR, GBP, JPY, CAD, AUD, HKD, TWD, SGD, MYR, KRW, CNY, THB, VND, IDR, PHP, INR). For currencies outside the map, set `currency: "USD"` and add a note in `vendorNotes`: `"Original currency <ISO>; converted to USD using <rate> on <date>"`.

     **NEVER drop a quote due to non-USD currency** — that was a documented framework gap (6 of 9 needs_review on the 13:30 historical recovery tick were Schukat EUR / Solsta GBP). The map handles it.

  (f) **Verifier disagreement on OPTIONAL fields → reconcile, don't bounce.** Per step 3.7's reconciliation rule, disagreements on optional fields (dateCode, packaging, rohs, coo) get resolved by re-reading source and picking the supported value. Only required-field disagreements (mpn, qty, cost, vendor) warrant needs_review.

  (g) **Vendor ambiguous on Type 2 (name-only) → name-based fuzzy plus context.** When Type 2 body has a short vendor name with multiple BP matches (e.g., "Poplar" → multiple "Poplar" BPs in OT), use any context from the body (geographic hints, prior quotes from the same broker compiled by the same operator) to pick. If still ambiguous, route `clarify_vendor` per step 3.8 — but with all candidate BPs in the payload so the operator's reply directly disambiguates.

**The rule:** if any of (a)-(g) recovers a viable load path, take it. `needs_review` should be reserved for cases where required data is genuinely absent or genuinely ambiguous after all recoveries — not for the routine inconveniences (a)-(g) handle.

**ALWAYS include `investigation_summary` on escalation payloads.** When routing any escalation action (`need_info_vendor`, `clarify_vendor`, `needs_vendor`, `needs_review`), include a payload field `investigation_summary` with a 1-2 sentence summary of what you tried before bouncing. Examples:

  - `"Pass 1 + Pass 2 returned 0 RFQs for MPN LM7805 across all active RFQs; no netCOMPONENTS URL in body; no subject RFQ#; cannot resolve."`
  - `"Domain @aedelectronics.com returned 2 active BPs after filtering Prohibited (1000005); activity tiebreaker shows AED Inc (140 active VQs) vs Acme Distrib (3 VQs, dormant since 2024) — picked AED."` *(this one would actually load, not escalate — example of how to phrase)*
  - `"Vendor name 'Howeher' fuzzy-matched 0 BPs in c_bpartner; mfr-aliases.json has no entry; investigation sub-Agent confirmed no name variant exists in OT."`

This breadcrumb makes the escalation auditable — was bouncing necessary, or did the agent prematurely surrender? The pattern is the new "show your work" requirement before escalation.

────────────────────────────────────────────
3.7.6 — SENDER outreach on escalations (VQ-specific override).
────────────────────────────────────────────

**VQ overrides the standing "info-requests go to operator, never the external sender" policy** (memory feedback_info_requests_go_to_operator_not_origin.md is excess/stockrfq policy; VQ is different).

Reason: VQ replies almost always go to a broker who CAN answer "what's the unit price?", "what's your legal company name?", "could you resend as Excel?" — the questions are about their own quote. The operator (Jake) still gets visibility via CC and the answer round-trips through vq@ for next-tick stitching.

**Routing the handlers apply automatically when `senderEmail` is in the payload:**
  - `need_info_vendor`, `clarify_vendor`, `needs_vendor`, `needs_review` → TO=senderEmail, CC=Jake
  - Reply-To = vq@ → reply round-trips, sidecar-stitch path picks it up
  - When `senderEmail` is missing/unparseable → handler falls back to operator-only (legacy behavior preserved)

**Your job:** ALWAYS include `senderEmail` on escalation payloads when you have it. The deepest non-Astute broker From: address — same value you'd pass on `load_vq`. Type 2 bulk summaries: the senderEmail will be the Astute compiler; still useful (Gopal/etc. can answer or forward).

**MULTI-VENDOR partial-load (the calibration target):** when one inbound email has quotes from N vendors and you can confidently load M of them but the remaining (N−M) have issues, DO NOT bounce the whole email. Load what you can via `load_vq` AND include `clarifications[]` in the same payload describing the outstanding asks per problematic vendor. The handler writes the clean VQs, then fires ONE consolidated email to the sender (CC Jake) covering all the loose ends in one go. The sidecar (kind=`partial_clarify`) holds the outstanding state for next-tick reply-stitching.

`clarifications[]` shape on the `load_vq` payload — each section MUST include `pendingQuotes[]` (the partial quote stubs you already extracted for that vendor, missing only the asks). The sidecar persists these so the reply-tick agent merges answers into structured stubs instead of re-extracting from email body — that's what scopes the merge to the unloaded vendors only and avoids double-writes against the already-loaded ones:

  ```
  clarifications: [
    { vendorLabel: "Acme Trading",
      asks: [
        "Confirm unit price for MPN XYZ-123",
        "Date code for SKU 456 (your message said 'recent stock', need numeric)"
      ],
      pendingQuotes: [
        { mpn: "XYZ-123", mfr: "TI", qty: 1000, vendorName: "Acme Trading",
          cost: null, dateCode: null, leadTime: "stock" },
        { mpn: "SKU-456", mfr: "STM", qty: 500, vendorName: "Acme Trading",
          cost: 0.85, dateCode: null }
      ]
    },
    { vendorLabel: "Bingo Brokers",
      asks: ["Quantity offered for MPN ABC?"],
      pendingQuotes: [
        { mpn: "ABC", mfr: "ADI", cost: 0.42, vendorName: "Bingo Brokers",
          qty: null, dateCode: "2024" }
      ]
    }
  ]
  ```

The handler also derives `loaded_vendors` automatically from the input `quotes[]` and persists it in the sidecar. On reply, the agent reads `pending_state.loaded_vendors` and NEVER includes those vendors in the merge load_vq.

Use `clarifications[]` for the partial-load multi-vendor case ONLY. For single-vendor failures or whole-email failures, use the normal escalation actions (`need_info_vendor` / `clarify_vendor` / `needs_vendor` / `needs_review`) — they also do sender outreach.

────────────────────────────────────────────
3.8 — VENDOR resolution.
────────────────────────────────────────────

For each quote, resolve the vendor BP. **The only role filter is `isactive='Y'`. Do NOT filter on `isvendor` / `iscustomer` / `isemployee` — many real vendors in Astute's OT are flagged as both vendor AND customer (broker-distributors we buy from AND sell to). A BP is a BP; role flags are billing/operational state, not "should we record a quote from them."** Suspended-vendor-type (1000004) is enforced at write time inside loadBulkSummary; the agent does not need to filter for it.

  Type 1 (broker email domain available):

    **Pass 1 — active BP via domain match on any registered contact:**
      SELECT bp.c_bpartner_id, bp.value, bp.name, bp.isactive, bp.chuboe_vendortype_id,
             vt.name AS vendortype_name, COUNT(u.ad_user_id) AS contact_count
      FROM adempiere.c_bpartner bp
      JOIN adempiere.ad_user u ON u.c_bpartner_id = bp.c_bpartner_id
      LEFT JOIN adempiere.chuboe_vendortype vt ON bp.chuboe_vendortype_id = vt.chuboe_vendortype_id
      WHERE bp.isactive='Y' AND u.isactive='Y'
        AND LOWER(SUBSTRING(u.email FROM POSITION('@' IN u.email)+1)) = '<vendor domain>'
        -- Auto-disqualify Suspended (1000004) and Prohibited (1000005) BPs.
        -- These are operator-flagged "do not buy from" — never auto-pick them,
        -- never include in clarify_vendor candidates. See shared/disqualified-vendor-types.js.
        AND COALESCE(bp.chuboe_vendortype_id, 0) NOT IN (1000004, 1000005)
      GROUP BY bp.c_bpartner_id, bp.value, bp.name, bp.isactive, bp.chuboe_vendortype_id, vt.name
      ORDER BY contact_count DESC, bp.created DESC
      LIMIT 5;

    **Activity-based tiebreaker** (when Pass 1 returns >1 row): pick the BP with the most recent VQ activity in the last 90 days, then highest active-VQ count. Run:
      SELECT vq.c_bpartner_id, COUNT(*) FILTER (WHERE vq.isactive='Y') AS active_vqs,
             COUNT(*) FILTER (WHERE vq.created >= NOW() - INTERVAL '90 days') AS recent_vqs,
             MAX(vq.created) AS latest_vq
      FROM adempiere.chuboe_vq_line vq
      WHERE vq.c_bpartner_id IN (<candidate ids from Pass 1>)
      GROUP BY vq.c_bpartner_id
      ORDER BY recent_vqs DESC NULLS LAST, active_vqs DESC, latest_vq DESC NULLS LAST;
    The top row is the operational BP. If recent_vqs and active_vqs are BOTH zero for all candidates → genuine ambiguity, fall through to clarify_vendor.

      The specific sender (e.g., `eric@inventorymp.com`) does NOT need to be in `ad_user`. ANY active user on the BP with the matching domain confirms vendor identity — IMP has 6 active contacts (Aaron, Greg, Jim, David, Tim) all on inventorymp.com; the BP matches even when Eric Ward himself is a newer contact not yet registered. Contact info is metadata, not a load gate.

    **Pass 2 — INACTIVE sweep, only if Pass 1 returned 0 rows:** Same query, drop the `isactive='Y'` filters from both `bp` and `u`. Distinguishes "BP exists but flagged inactive" from "truly new vendor". Look for "USE <searchKey>" patterns in inactive BP names — that's OT's standard "this BP is replaced by N" marker, and the operator may want to redirect.

  Type 2 (vendor name from body — no email):
    Fuzzy name match — `LOWER(bp.name) ILIKE '%<shortname>%' AND bp.isactive='Y'`. Common shortenings: Poplar → Poplar Technology, Howeher → Howeher Co. Limited.
    If 0 active hits, repeat without `isactive='Y'` for inactive sweep.

  Decision logic (applies to both Type 1 and Type 2):
    - Pass 1 returns exactly 1 → use it; pass `vendorSearchKey` on the quote object.
    - Pass 1 returns >1 → route `clarify_vendor` with `{candidates: [{bpartnerId, searchKey, name, isactive, reason}, ...], vendorName, vendorEmail, extracted: <quotes>, subject, outerFrom, senderEmail}`. keepsPending.
    - Pass 1 returns 0, Pass 2 returns ≥1 → route `clarify_vendor` with the inactive candidates (each `isactive: false` in the candidate object). The candidates' names often include `USE <N>` pointers — operator decides whether to reactivate, redirect to the pointer, or skip. Same payload shape as the ambiguous case; the email surfaces the inactive flag explicitly.
    - Pass 2 also returns 0 → route `needs_vendor` with `{vendorName, vendorEmail, extracted: <quotes>, subject, outerFrom, senderEmail}`. keepsPending — sidecar waits for Jake to add the vendor; next tick auto-detects and loads.

  **Reminder:** include `senderEmail` on every escalation payload per step 3.7.6 so the handler can outreach to the broker (CC Jake) instead of operator-only.

The loadBulkSummary library also handles vendor resolution server-side — passing `vendorName` alone (no `vendorSearchKey`) lets it fall back to name fuzzy match. But the agent should resolve up front when the data is available, so it can route clarify/needs_vendor for ambiguity rather than having the writer drop the quote silently as `VENDOR_NOT_FOUND`.

────────────────────────────────────────────
3.9 — DEDUP check (pre-write idempotency, per-RFQ).
────────────────────────────────────────────

The check runs **once per target RFQ** (primary + all secondaries from step 3.6.4). Multi-write means RFQ A might already have this quote (dup_skip on A) while RFQ B is fresh (write to B). Each target RFQ is independent.

For each (rfqSearchKey, quote) pair, check for an existing active VQ with the same shape within 30 days:

  SELECT vq.chuboe_vq_line_id, vq.created, bp.name as vendor
  FROM adempiere.chuboe_vq_line vq
  JOIN adempiere.chuboe_rfq_line rl ON vq.chuboe_rfq_line_id = rl.chuboe_rfq_line_id
  JOIN adempiere.chuboe_rfq r ON rl.chuboe_rfq_id = r.chuboe_rfq_id
  JOIN adempiere.c_bpartner bp ON vq.c_bpartner_id = bp.c_bpartner_id
  WHERE vq.isactive='Y' AND r.value='<rfqSearchKey>'
    AND vq.c_bpartner_id=<vendorBpartnerId>
    AND UPPER(vq.chuboe_mpn)=UPPER('<mpn>')
    AND vq.qty=<qty>
    AND vq.cost=<cost>
    AND vq.created >= NOW() AT TIME ZONE 'America/Chicago' - INTERVAL '30 days';

Routing decision after dup check across all targets:
  - If EVERY (rfq, quote) pair is a dup → route `dup_skip` with the first existing `existingVqId`. Move to Duplicates. Don't load_vq.
  - If SOME targets are dup and others are fresh → route `load_vq` with `rfqSearchKey` + `secondaryRfqSearchKeys` set to ONLY the non-dup target RFQs. The handler writes to whatever's left. Breadcrumb the dup-skipped RFQs in the route output's `notes` field.
  - If NO duplicates → route `load_vq` with the full target list.

────────────────────────────────────────────
3.11 — POST-WRITE check.
────────────────────────────────────────────

Inspect the route response:
  - `skipped` non-empty → some quotes hit VENDOR_NOT_FOUND / NO_MPN_MATCH / VENDOR_SUSPENDED. If you resolved vendors in 3.8 these should be rare — log per-quote and continue.
  - `failed` non-empty → writer threw on some quotes. Surface in your summary; don't reroute (the email already moved to Processed).
  - `gaps` non-empty → some RFQ lines have no coverage from this batch. Note in summary — next-action signal for the sourcing team.
  - `written > 0` and no critical failures → success. Email moved to Processed by the handler.

If `written === 0 && skipped + failed === quotes.length` → consider routing `needs_review` instead next time (currently the email already moved to Processed; flag this in the summary).

═══════════════════════════════════════════════════════════════════════════════
STEP 4 — Summary.
═══════════════════════════════════════════════════════════════════════════════

At the end, print a single-line summary:

  VQ-LOADING-AGENT SUMMARY: processed=<N> load_vq=<N> rfqs_written=<N> vqs_written=<N> vqs_skipped=<N> vqs_failed=<N> need_info_vendor=<N> clarify_vendor=<N> needs_vendor=<N> needs_review=<N> no_bid=<N> not_vq=<N> dup_skip=<N> outbound_pending=<N> errors=<N>

`load_vq` counts route calls (one per email). `rfqs_written` is the total distinct RFQs that received at least one VQ across all routes — distinct from load_vq because multi-RFQ fan-out can push one email into N RFQs. `vqs_written` sums across all RFQs.

If a single message hits a transient error (loadBulkSummary throws, attachment unreadable, vendor lookup 5xx), log the error in your output and continue with the next message. Do not abort the batch on per-message failure.

NEVER include --dry-run on any route call. This is the production schedule.
